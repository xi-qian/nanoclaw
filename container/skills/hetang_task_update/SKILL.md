# hetang_task_update（单独项目任务运维 Skill）

本 Skill 专门用于指导 Agent（飞书机器人）如何对**已经创建**的"单独项目管理"文档中的"任务"进行工作汇报、查询、修改等运维操作。

> **硬门禁（选择目标）**：当用户要求"更新任务"、"汇报工作（如：已完成XX工作）"等运维操作时，Agent **必须**首先读取 `memory/feishu_bitable_links.md` 的「历史记录」区，解析每条中的 **`名称:`**（或等价字段），**去掉首尾空格后去重**。
>
> - **若去重后存在 2 个及以上不同的文档名称**，或 **「历史记录」区有效行数 ≥ 2**（含**同名但不同链接**的多条记忆，如旧版/新版各一行）：一律视为**须由用户选定目标文档**，**必须**逐条列出（建议每条显示：**名称** + **日期或链接尾段** 以便区分），并提问：「请问本次要操作哪一条项目文档？」**在用户明确点选某一条（或粘贴完整链接 / app_token）之前，禁止**查询任务表、禁止调用任何写表 MCP、禁止推断上下文。**
> - **绝对禁止**：根据用户汇报里的**医院名、科室名、疾病类型、任务用语**等与某条记忆名称「看起来像」就**自动选中**某个项目（例如从「血液科」推断「血液病医院 NK 项目」）。
>
> - **若去重后仅有 1 个文档名称**：仍须**先发一条消息**写明：「将操作项目文档：〈名称〉（可附链接）」，请用户**确认或回复「是」**后再进入任务操作；**禁止**零确认直接查表。

> **数据源约定**：运维以**用户当次选定**的 Base（`app_token`）为准；`memory/feishu_bitable_links.md` 仅作索引。若记忆与项目总表「项目管理文档链接」不一致，**以用户明确指定或飞书总表为准**。

## 项目阶段与里程碑定义（参考资料）

项目管理共分为 8 个阶段，括号内为该阶段的核心里程碑任务：
1. **第一阶段**：前期拓客·BD开发（合作意向）
2. **第二阶段**：需求交流·PI多轮沟通（需求会议纪要）
3. **第三阶段**：内部立项评估（立项审批表）
4. **第四阶段**：材料准备·20+ 模块逐一撰写（研究方案、资质文件、医院模板类文件）
5. **第五阶段**：外部立项·医院三线并行审批（科学评估、伦理审批、签订合同）
6. **第六阶段**：临床实施·患者运营（临床中心启动、细胞准备与质控、首例入组、完成入组、关闭临床）
7. **第七阶段**：数据分析·结题发表（数据统计）
8. **第八阶段**：资料归档与关闭（资料归档与关闭）

**阶段父行任务名称（与 `single_project` 模板初始父记录一致，用于辅助识别）**：`前期拓客·BD开发`、`需求交流·PI多轮沟通`、`内部立项评估`、`材料准备·20+ 模块逐一撰写`、`外部立项·医院三线并行审批`、`临床实施·患者运营`、`数据分析·结题发表`、`资料归档与关闭`。

## Agent 执行原则

### 1. 强制读取记忆文件与选择文档
- 永远先读取 `memory/feishu_bitable_links.md` 的「历史记录」区。
- 如果没有记录：告知用户请先创建。
- **须用户选定文档的判定**（满足**任一**即不可自动定项）：**历史记录有效行数 ≥ 2**；或 **`名称:` 去重后 ≥ 2 个不同名称**。
- **上述情况**：逐条列出供选择（与硬门禁一致），**等待用户明确点选某一条**，**禁止关键词推断**。
- **历史记录恰好 1 行**：仍须**先展示该条「名称」（及可选链接）并请用户确认**后再操作；不得跳过确认直接访问任务表。

### 2. 确定目标与表信息
- 默认操作的表格为"任务"表。
- 通过飞书 MCP 工具获取该 `app_token` 下的 `table_id`（名称为"任务"）。
- **日期类字段名**须与表结构一致，荷塘单独项目模板为：**`开始时间`**、**`截止时间`**、**`完成时间`**（`date` 类型）。若实际 Base 字段名不同，**以 list_fields / MCP 返回的字段名为准**。
- **成员表 — 申报人自动建档（强制）**：在填写任务的「负责人」「任务执行人」前，须用 MCP 在「成员」表查询当前**申报工作人员**（当前对话中的汇报人）是否已有记录（通常按 **`账号`**（`user` 类型）匹配）。
  - **若不存在**：**必须**先通过 MCP 在「成员」表 **新增一条记录**：**`账号`** = 该申报人；**`角色`** 默认设为 **`客户负责人`**（`multi_select`，选项名须与表中已有选项一致）。创建成功后取得该成员行的 **`record_id`**，再用于任务的 duplex_link。
  - **禁止**仅因「成员里没有此人」就中断流程或只让用户去手工添加；**除非** MCP 无法完成成员表的创建/写入，再按 §7 报错说明。

### 3. 阶段父记录与子任务的判定（新增子任务前必读）

- **阶段父记录（父行）**：满足以下**任一**即可视为阶段根行（用于第3步筛选「目标阶段」）：
  1. 字段 **`父记录 2`** 为空、未填或**未指向本表其他任务记录**；
  2. 或 **`任务名称`** 与上文「阶段父行任务名称」列表中**某一项完全一致**。
- **子任务**：**`父记录 2`** 已指向某条阶段父记录或其他任务记录的 `record_id`。**新增汇报任务必须作为子任务**，`父记录 2` = `["<目标阶段父记录的 record_id>"]`，**禁止**与阶段父行同级（禁止顶层新行、禁止漏填 `父记录 2`）。

### 4. 状态推断（第2步与第3步共用）

根据用户话术推断要写入「状态」单选（**仅允许**：`未开始`、`进行中`、`已完成`、`已停滞`）：
- **已完成** / **做完** / **搞定** / **结束** 等 → `已完成`
- **开始** / **启动** / **进行中** / **在做** 等 → `进行中`
- **暂停** / **卡住** / **停滞** 等 → `已停滞`
- **还没开始** / **未启动** 等 → `未开始`
- **无法可靠推断**时默认 `已完成`（与「汇报完成工作」场景一致）。

**禁止**把用户表述的**完成时间**写入 **`开始时间`**。

### 5. 工作汇报处理逻辑（核心业务规则）

> **⚠️ 前置门禁（绝对禁止跳过）**：每次处理工作汇报前，Agent **必须**严格遵守「1. 强制读取记忆文件与选择文档」的规定。即使上下文中有最近操作过的项目文档，**也必须**重新读取 `memory/feishu_bitable_links.md` 并**按名称去重完成项目选择或确认**。**禁止**凭上一轮对话或用户话术中的科室/医院等词**自动绑定**某个 `app_token`。

主流程为：**第0步（如适用）→ 第1步 → 分支（第2步 或 第3步）**。分支内子步骤须顺序执行，**不得跳步**。

**第0步：多阶段 / 整阶段批量完成（强制优先判断）**

当用户汇报**同时涉及两个及以上「项目阶段」且均表示已完成**（例如：「第一阶段与第二阶段的工作内容都已完成」），或表述为**某一/多个阶段内全部工作已完成**、又**未指向单条具体任务名称**时，必须先走本步，**不得**直接进入第1步用「单任务名」硬匹配后默写日期。

- **禁止**：在用户未提供**每条受影响任务**（或用户明确认可的**统一日期规则**）的 **开始时间、完成时间** 前，**批量**把多条任务的日期默填为「今天」或主观猜测。
- **必须**：
  1. 用 MCP 列出受影响阶段下的**全部相关任务**（该阶段父行下的子任务、里程碑等；以 `项目阶段` 与 `父记录 2` 层级为准）。
  2. 向用户展示清单，并**明确询问**：请为**每一条任务**分别提供 **开始日期** 与 **完成日期**（可请用户按「任务名 — 开始 — 完成」逐条回复）；或请用户声明**统一规则**（例如：「以下列出的所有任务均为 YYYY-MM-DD 开始、YYYY-MM-DD 完成」且覆盖清单内每一项）。
  3. 信息未补齐前**不得**批量更新；补齐后按 §6 计算时间戳，再逐条（或按用户统一规则）更新「状态」及 **开始时间 / 完成时间**。
  4. 批量更新完成后，对**本次涉及的所有阶段父记录**执行下文 **「添改任务后的阶段时间汇总」**。
- **例外**：用户**同一条消息里**已写清每一项的日期，或已写清覆盖全部所列任务的统一开始日、完成日，可直接进入更新，**无需再问**。

**第1步：查找任务**
1. **优先**：在「任务」表按 **`任务名称`** 与用户汇报内容**精确匹配**。
   - **若仅唯一命中**但该任务所属**阶段父记录**已为 **`已完成` / `已停滞`**，且用户表述像**当前工作汇报**（非「修正历史数据」），则**不得直接进入第2步**：须先说明该任务处于**已结束阶段**，询问用户是**仍更新本条**还是**按第3步新建到当前有效阶段**。
2. **若无结果**：再按**包含关系**（汇报文本包含任务名，或任务名包含汇报关键词）列**候选**前，须先为每条可能命中解析 **所属阶段父记录**（沿 **`父记录 2`** 上溯至 §3 定义的**阶段根行**），读取该父行的「状态」「项目阶段」。
   - **与「选定阶段」规则对齐（强制）**：本 Skill 规定**新增子任务**仅允许挂在 **阶段父记录状态为 `进行中` 或 `未开始`** 的父行下（见第3步）。因此，当用户汇报为**当前工作**（如「已完成与××的讨论」）而**未明确点名**要改某条**历史归档任务**时：
     - **禁止**仅列出**落在「阶段父记录已为 `已完成` 或 `已停滞`」子树内**的任务作为**唯一或首推**候选，而不说明阶段已结束、不给出「新建到当前有效阶段」的选项。
     - **必须**：**优先展示**候选中那些所属阶段父记录为 **`进行中` / `未开始`** 的任务；若仍要展示已结束阶段内的相似任务（如「立项第一轮讨论」在**已完成**的第三阶段下），**必须标注**：「该任务位于第X阶段，**该阶段父记录已为已完成**；若您登记的是**当前项目进展**，请选 **新增子任务**（将按第3步挂到**最早未结束阶段**），或明确回复仍要更新本条历史任务。」
     - **若包含匹配仅命中已结束阶段子树、且用户未声明要改历史任务**：**默认进入第3步**（在最早 `进行中`/`未开始` 阶段父记录下新建子任务），可同时**附**「相似历史任务」供用户改口选用，但**不得**把历史任务当作默认唯一路径。
3. **候选条数**：若**多条**候选，**必须**请用户**指定一条**；若**零条**（在按上款过滤后仍无合适「当前阶段」命中），进入**第3步**。
4. **唯一匹配** → **第2步**；**用户选定一条** → **第2步**（若选定任务位于**已完成阶段**，更新前须已获用户**明确确认**「就是要改这条历史任务」）。

**第2步：找到任务时**
依次执行：
1. 将「状态」更新为 **§4 推断出的目标状态**（勿写死仅为「已完成」）。
2. **补充字段（禁止跳过）**：
   - **完成时间**：若本次属于 **第0步「多阶段批量完成」** 流程，**不适用**「未提及则默认今天」；须已按第0步收集到该条的完成日期后再写入。否则：若用户明确给出完成日期 → 用 §6 脚本算毫秒戳写入；未提及且状态为 `已完成` → **今天**（CST）的毫秒戳；若状态不是 `已完成`，一般**不填完成时间**（除非用户同时声明完成日）。
   - **开始时间**：若本次属于 **第0步** 流程，须已按第0步收集到该条的开始日期后再写入。否则：若该任务尚无开始时间，**必须**提问「请提供该任务的开始日期」；用户回复后用 §6 写入。**禁止**把「完成」相关日期写入开始时间。
   - **任务执行人** / **负责人**：若为空，**先按 §2「成员表 — 申报人自动建档」** 确保申报人在「成员」表中存在（无则自动新增，`角色` 默认 **客户负责人**），再将其 **`record_id`** 填入 MCP 要求的 **duplex_link 形态**（**以 MCP 工具 schema 为准**）。**禁止**随便填文本名。
3. **可选（阶段收口）**：若该任务有父阶段，且经 MCP 列出该父阶段下**所有子任务**后，**全部**状态均为 `已完成`，则可将**阶段父记录**的「状态」更新为 `已完成`；若无法可靠枚举子任务，则跳过本步并可在回复中说明。
4. **阶段时间回填**：若本次更新涉及某条**阶段父记录**下的任务，**必须**对该阶段父记录执行下文 **「添改任务后的阶段时间汇总」**（仅在该父行的对应列为空时回填）。

**第3步：未找到任务时（新增子记录）**

> **强制规则（层级）**：新增任务**必须**挂在当前目标阶段父记录之下（子任务），**禁止**与阶段父行同级；**禁止**省略或留空 **`父记录 2`** 导致落在表顶层。

按顺序严格执行：
1. **筛选可用父记录**：在**阶段父记录**中，筛「状态」为 `进行中` 或 `未开始` 的行；**禁止**选用 `已完成` / `已停滞` 的父行作为目标阶段。
2. **确定目标阶段**：将筛出的父记录按「项目阶段」正序（第一→第八阶段）排列，取**第一个**为**目标阶段父记录**。
3. **必填项提问**：提问「请提供该任务的开始日期」，用户回复后再创建（开始时间用 §6 计算）。
4. **创建子记录**：
   - **`父记录 2`**：`["<目标阶段父记录的 record_id>"]`（**不是**其他子任务、**不是**「父子任务」字段名，除非表结构仅有该别名——以实际字段名为准）。
   - **字段设置**：`任务名称` = 汇报任务名；`状态` = §4 推断值；`项目阶段` = 与目标父行一致；**开始时间** = 用户回复日期（§6）；**完成时间** = 用户若在汇报中声明则写入，否则若状态为 `已完成` 则默认今天（§6）；**负责人** / **任务执行人** = 先按 **§2** 确保申报人已入「成员」表后，使用该成员行的 **record_id**（duplex_link 形态以 MCP schema 为准）。
5. **更新阶段状态**：将目标阶段父记录的「状态」更新为 `进行中`（若已为 `进行中` 可幂等跳过）。
6. **阶段时间回填**：对本次挂接的**目标阶段父记录**，执行下文 **「添改任务后的阶段时间汇总」**（仅在该父行的对应列为空时回填）。

**添改任务后的阶段时间汇总（强制）**

在**第0步批量**、**第2步**或**第3步**中，对任务表完成**新增或更新**后，对**受影响的每一条阶段父记录**须执行本汇总（可与「更新阶段状态」「阶段收口」在同一轮 MCP 流程中完成）：

- **范围**：「该阶段下所有任务」指 **`父记录 2` 以该阶段父记录为祖先的整棵子树**内全部任务行（须通过 MCP 按层级展开：先列直接子项，再对仍有下级的记录继续向下查询，直至收齐；若 MCP 能力受限，至少覆盖**直接子任务**，并在回复中说明是否未纳入更深层级）。
- **阶段父行的列名**：与任务表一致时，荷塘模板为 **`开始时间`**、**`完成时间`**（用户口语中的阶段「结束时间」在无单独列时**对应 `完成时间`**）。若实际 Base 另有「结束时间」等列，**以 list_fields / MCP 为准**。
- **规则（仅当对应列为空时才写入，已有值不得覆盖）**：
  1. 若阶段父记录的 **`开始时间`** 为空：在子树任务中收集所有**非空**的 **`开始时间`**（毫秒戳或可比较日期），取**最小值**写入阶段父行的 **`开始时间`**（写入形态遵循 §6 / MCP schema）。若子任务**全无**有效开始时间，则**不填**。
  2. 若阶段父记录的 **`完成时间`**（或上条认定的阶段「结束」汇总列）为空：在子树任务中收集所有**非空**的 **`完成时间`**，取**最大值**写入该列。若子任务全无有效完成时间，则**不填**。
- **同一轮操作影响多个阶段**（如第0步）：须对**每个**涉及的阶段父记录**分别**做上述 min/max 汇总。

### 6. 时间戳计算规则（强制要求）

- **`开始时间` / `截止时间` / `完成时间`** 写入飞书的值形态：**以所用 MCP 工具（如 `update_bitable_record`）的 JSON schema 为准**（可能是裸毫秒整数，或 `{"date": 毫秒戳}` 等）。调用 MCP **前须阅读该工具的 descriptor**，按 schema 包装后再写入。
- 日期对应的**日历日**换算为 **CST（UTC+8）当日 0 点**的毫秒时间戳时，**禁止心算**，必须用脚本：

```python
from datetime import datetime, timezone, timedelta

cst = timezone(timedelta(hours=8))

def date_to_ts(year, month, day):
    d = datetime(year, month, day, 0, 0, 0, tzinfo=cst)
    return int(d.timestamp() * 1000)

# 示例：print(date_to_ts(2026, 4, 2))
```

### 7. 执行运维操作（MCP 与 Python 边界）

- **对飞书多维表格**：记录的查询、创建、更新**必须**通过 MCP 工具（如 `CallMcpTool` 调用 `list_bitable_records`、`update_bitable_record`、`add_bitable_records` 等）；**禁止**用 Python/Shell **直接请求**飞书 OpenAPI。
- **允许例外**：仅用 Python/Shell 做**本地计算**（时间戳、候选匹配、字符串处理），**不得**携带 `app_secret` 或替代 MCP 写表。
- **报错**：若无 MCP 能力完成某步，**立即停止**并回复：「抱歉，当前系统未提供执行此操作的 MCP 工具，无法完成任务更新。」
