```
用户消息
    │
    ▼
┌─────────────────┐
│  Channel 层     │  onMessage → storeMessage() → SQLite

└────────┬────────┘
         │
         ▼

         │
         ▼
┌─────────────────┐
│  群组队列        │  GroupQueue.enqueueMessageCheck()

└────────┬────────┘
         │
         ▼
┌─────────────────────────────────────────────────────────────┐
```

## 瓶颈分析

### 不是瓶颈
- **容器启动**: 5ms，几乎瞬间
- **SDK 初始化**: Session 复用后仅 9ms

### 主要瓶颈
- **API 响应时间**: 当前使用 `open.bigmodel.cn` 代理，响应不稳定
  - 简单回复需要 12.9 秒 TTFB
  - 网络搜索请求需要 3-18 秒

### API 端点测试

```bash
# 代理 API 响应速度
curl -w "TTFB: %{time_starttransfer}s\n" https://open.bigmodel.cn/api/anthropic
# 结果: 0.15 秒 (连接正常，但推理慢)
```

## 优化建议

| 优先级 | 方案 | 预期收益 |
|--------|------|----------|
| 高 | 切换到官方 API `api.anthropic.com` | 减少 50-80% API 延迟 |
| 中 | 使用更快的模型 (如 claude-3-haiku) | 减少推理时间 |
| 低 | 消息轮询改为事件驱动 | 减少平均 1 秒延迟 |

### 关键发现

#### ✅ 容器复用机制已实现

NanoClaw 使用智能容器复用策略：

```
首次查询: 启动新容器 → 完成后保持运行 (IDLE_TIMEOUT=30min)
后续查询: 通过 IPC 文件传递消息 → 复用同一容器 → 无需启动
```

**容器日志证据:**
```
[22:54:00.426] [+1ms] [perf] main: container started      # 第一次
[22:58:03.351] [+242926ms] Piping IPC message into active query  # 第二次，无启动日志
```

#### ⚠️ 性能瓶颈确认

**容器启动不再是瓶颈！**

第二次测试的性能分布：
```
API TTFB    ████████████░░░░ 75%
轮询延迟    ███░░░░░░░░░░░░░ 18%
其他        █░░░░░░░░░░░░░░░  7%
```

**主要瓶颈:**
1. **智谱 API TTFB** (5.8s) - 占总时间的 75%
2. **轮询延迟** (1.4s) - 平均等待 1-2 秒检测新消息

#### 容器启动时间分析

**首次启动 3.5 秒的原因:**
- Docker daemon 响应: ~100-500ms
- 镜像层加载: ~500-2000ms (首次或缓存未命中)
- 文件系统挂载: ~500-1000ms (6个 volume mount)
- 进程启动: ~200-500ms

**服务不重启时的表现:**
- Docker 会缓存镜像层
- 但每次 `docker run` 仍需创建新容器实例
- **容器复用通过 IPC 实现，完全避免启动开销**

### 容器生命周期

```
查询 1: spawn 容器 → 执行 → 保持运行 (30分钟)
查询 2: IPC 消息 → 复用容器 → 执行 → 保持运行
...
30分钟无活动: 容器超时清理
查询 N: spawn 新容器 → 循环
```

**关键配置:**
- `IDLE_TIMEOUT = 1800000` (30分钟) - 容器空闲后自动清理
- `IPC_POLL_INTERVAL = 1000` (1秒) - IPC 消息检测频率
